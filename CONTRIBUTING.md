# Contributing

This repo builds every Wolf app image under `images/<name>/` and publishes it to GHCR. The pipeline is
convention-based, so adding an image needs zero CI config.

## CI/CD

[`.github/workflows/build-images.yml`](.github/workflows/build-images.yml) builds **every image in
parallel** and pushes to GHCR. It runs automatically:

- **on push to `main`** — only the images whose folder changed (or all images if a shared CI file
  changed). This is the publishing run: it pushes to GHCR and signs.
- **on pull request** — the same build runs to **validate** the change, but does **not** push or sign,
  so a Renovate bump that breaks the build is caught before it can merge;
- **manually** (`workflow_dispatch`) — optionally limited to a comma-separated list of image names;
  ignores the change detection and builds exactly what you ask for.

Each image is tagged with:

- **`edge`** — rolling, every default-branch build;
- **`<version>`** — the upstream version pinned in the Dockerfile, when present (read from the
  `# renovate:`-annotated `ARG <APP>_VERSION=` line).

Images are **signed keyless with [cosign](https://docs.sigstore.dev/)** (Sigstore + GitHub OIDC — no
private key to manage). Verify a pull with:

```bash
cosign verify ghcr.io/rubasace/soh:edge \
  --certificate-identity-regexp '^https://github.com/rubasace/gow/' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
```

The matrix is generated by [`.github/scripts/build-matrix.sh`](.github/scripts/build-matrix.sh), which
discovers images, decides which ones to build for the current event, and extracts each version. It's
runnable locally (`GITHUB_EVENT_NAME=workflow_dispatch bash .github/scripts/build-matrix.sh`).

### Version bumps (Renovate)

[`renovate.json5`](renovate.json5) keeps everything current automatically:

- **App versions** — the `# renovate:`-annotated `ARG <APP>_VERSION=` pins (SoH, 2Ship, …). When
  upstream releases, Renovate opens a PR bumping the pin.
- **Base image** — `pinDigests` pins `base-app:edge` to a digest, so Renovate also tracks the rolling
  base and bumps the digest when upstream republishes `edge` (the only way to follow a non-versioned
  tag). The tag stays for readability; the digest is what builds.

**Auto-merge policy:** patch/minor app releases and base-image digest bumps **merge themselves** once
the PR build is green — day-to-day updates need no attention. **Major** bumps are left as a PR to
review by hand (they can change the release asset layout or force a re-extraction). Merging any of them
to `main` triggers the publishing build. Adjust the policy in the `packageRules` block of
[`renovate.json5`](renovate.json5).

## Adding a new image

Zero CI config — the pipeline is convention-based:

1. Create `images/<name>/` with a `Dockerfile` (and whatever `scripts/`, `configs/`, `wolf/` it needs).
   The folder name **is** the GHCR repository name, so keep it lowercase and registry-safe.
2. To get a version tag in addition to `edge`, pin the upstream version with a Renovate-annotated build
   arg — the existing images follow this exactly:

   ```dockerfile
   # renovate: datasource=github-releases depName=<owner>/<repo>
   ARG MYAPP_VERSION=1.2.3
   ```

   The arg name only has to end in `_VERSION`. An image with no such pin still builds and gets the
   rolling `edge` tag.
3. Open a PR / push to `main`. The new folder is detected and built automatically; no change to the
   workflow is needed.

Add a `README.md` in the image folder explaining how to use it, and link it from the table in the
root [README](README.md).
