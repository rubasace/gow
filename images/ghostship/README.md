# Ghostship (Super Mario 64) for Wolf (Games on Whales)

Wolf image that **packages the official Linux release** of [Ghostship](https://github.com/HarbourMasters/Ghostship)
— the HarbourMasters *Super Mario 64* PC port, built on the same libultraship engine as
[`soh`](../soh) and [`2s2h`](../2s2h) — with no compiling of the game. The game-specific bits are:
the binary is `Ghostship`, the baked archive is `ghostship.o2r`, the ROM archive is `sm64.o2r`, and
there's a single ROM archive (no Master Quest equivalent). State lives in three places:

- **the binary** is baked into the image, matched to the version;
- **settings and saves** stay **per-user** in the home Wolf auto-mounts;
- **the ROM** and the derived **`sm64.o2r`** live in a **shared folder** you manage on the host.

Updating is just bumping the image tag and rebuilding — saves are untouched and the ROM isn't
re-extracted, except on a major version jump (handled automatically, see [Updating](#updating)).

## Where things live

| Lives in | What | Persistence |
| - | - | - |
| **Image** (`/opt/ghostship`, read-only) | `Ghostship` + libs + `ghostship.o2r` + `config.yml` + `assets` + the `torch` extractor | ships with the release |
| **Wolf home** (`/home/retro`, auto) | `ghostship.cfg.json`, saves, `mods/` | **per-user** (Wolf mounts it per profile) |
| **Shared** (`/mnt/wolf/shared/ghostship` → `/roms`) | ROM `.z64`, common `sm64.o2r`, `mods/` | **shared** across profiles, you manage it |

You only ever provide the **ROM `.z64`**. The `sm64.o2r` is derived from it — either drop one in
the shared folder yourself (e.g. one generated by your desktop copy) or let the container generate
it on first launch.

## Usage

1. **Build** on the Wolf host (so the image stays local — Wolf launches containers through the
   host's docker socket), from this folder:
   ```bash
   docker build -t ghostship:wolf .
   # pin a version: docker build --build-arg GHOSTSHIP_VERSION=2.0.0 -t ghostship:wolf .
   ```
2. **Shared folder + ROM** on the host:
   ```bash
   sudo mkdir -p /mnt/wolf/shared/ghostship
   sudo cp "your-rom.z64" /mnt/wolf/shared/ghostship/   # MUST be US or JP (see Supported ROMs)
   # optional: drop an already-generated sm64.o2r here to skip generation entirely
   # to let the container generate + share the sm64.o2r itself, make the folder writable:
   sudo chown 1000:1000 /mnt/wolf/shared/ghostship
   ```
3. Paste `wolf/config.snippet.toml` into Wolf's `config.toml` and restart Wolf. It shows up in Moonlight.
4. **First connection:** if you didn't provide an `sm64.o2r`, the container builds it from the ROM
   and shares it back. This is **hands-free** — it does NOT use Ghostship's in-app extractor (which
   on Linux is gated behind a blocking ImGui popup and a file picker a gamepad can't dismiss);
   instead it runs the bundled **Torch** asset processor directly. You see the desktop for a moment
   while it extracts, then the game starts. Settings and saves go to your per-user home.

## Supported ROMs

Ghostship accepts **only US and JP** dumps, in **`.z64`** format (from upstream's `config.yml` /
`readme.txt`):

| Version | sha1 |
| - | - |
| **US** | `9bef1128717f958171a4afac3ed78ee2bb4e86ce` |
| **JP** | `8a20a5c83d6ceb0f0506cfc9fa20d8f438cafe51` |

**PAL/Europe and the Shindou re-release are not supported** and extraction will fail on them.
Verify your dump's sha1 at <https://www.romhacking.net/hash/>. If it's a `.n64`/`.v64` (byte-swapped)
dump, convert it to `.z64` first. Torch's output streams to the container logs; if extraction fails
the container logs the error and **exits** (no silent hang on a GUI prompt) — fix the ROM and reconnect.

## Updating

Bump the image tag and rebuild. Whether the ROM is re-extracted depends on the version jump:

- **`X.Y.Z` patch / feature** → the common `sm64.o2r` is reused. Just rebuild and redeploy.
- **major (`X.0.0`)** → the `sm64.o2r` may be outdated; the container detects it (via the version
  marker) and **regenerates it automatically and hands-free** from the ROM, then shares the fresh
  one back. You still only provide the `.z64`. (Needs the shared folder writable; on a read-only
  mount it warns instead.)

The version pin (`ARG GHOSTSHIP_VERSION` in the Dockerfile) is kept up to date by Renovate — see
[CONTRIBUTING](../../CONTRIBUTING.md).

## Settings and mods

- **Settings:** the image ships a default `ghostship.cfg.json` (fullscreen, OpenGL, a pre-mapped
  controller), seeded into a profile **only on first launch**. After that the user's own changes
  persist per-profile.
- **Mods:** drop `.o2r` / `.otr` files in `/mnt/wolf/shared/ghostship/mods/`; they're symlinked
  into each profile's `mods/` on launch (shared once, not duplicated per profile). Ghostship loads
  every archive it finds there. Note Ghostship pops an in-app SDL "untrusted author" warning for
  unsigned mods — fine with a keyboard, but a Moonlight gamepad can't dismiss it, so only add mods
  you're willing to confirm out-of-band.

## Notes

- **Why Torch is built into the image:** unlike `soh` (which can extract by passing the ROM as an
  argument) and `2s2h` (which bundles a `ZAPD.out` CLI), Ghostship's Linux release ships no
  extractor binary and its only in-app extraction path is GUI-gated. So the image builds the
  standalone **Torch** CLI at the exact commit Ghostship pins as its submodule for this version
  (resolved from the release at build time), guaranteeing the extractor matches the bundled
  `config.yml` and asset definitions. It runs once, on first launch, to generate `sm64.o2r`.
- **NVIDIA:** keep `/dev/nvidia*` in `GOW_REQUIRED_DEVICES` (already in the snippet); on Intel/AMD
  base-app's Mesa is enough.
- **Not tested end-to-end** (no Docker in the authoring environment): built against the official
  2.0.0 release, the `gow` base-app contract, the pinned Torch commit, and inspection of the AppImage.
- Implementation details (build stages, the hands-free extraction flow, how the `sm64.o2r` is
  bridged in) live in the commented `Dockerfile` and `scripts/`.
