# gow — Games-on-Whales / Wolf images

A collection of [Wolf](https://games-on-whales.github.io/) app images, each built
from `images/<name>/` and published to the **GitHub Container Registry (GHCR)**.

| Image | GHCR | Tags |
| - | - | - |
| Ship of Harkinian | `ghcr.io/rubasace/shipofharkinian` | `edge`, `<soh-version>` (e.g. `9.2.3`) |

More images (2Ship2Harkinian, OpenGoal, a custom EmulationStation, …) plug into
the same pipeline — see [Adding a new image](#adding-a-new-image).

## Pulling

```bash
docker pull ghcr.io/rubasace/shipofharkinian:edge      # latest build of main
docker pull ghcr.io/rubasace/shipofharkinian:9.2.3     # pinned to that SoH release
```

`edge` always points at the most recent successful build of `main` (mirrors the
Games-on-Whales naming convention). The numeric tag is the exact upstream version
baked into the image, so deployments can pin and updates are explicit.

## CI/CD

[`.github/workflows/build-images.yml`](.github/workflows/build-images.yml) builds
**every image in parallel** and pushes to GHCR. It runs automatically:

- **on push to `main`** — only the images whose folder changed (or all images if a
  shared CI file changed);
- **weekly** (cron) — rebuilds everything so `edge` and the version tags pick up
  base-app / OS security updates even when this repo is untouched;
- **manually** (`workflow_dispatch`) — optionally limited to a comma-separated list
  of image names.

Each image is tagged with:

- **`edge`** — rolling, every default-branch build;
- **`<version>`** — the upstream version pinned in the Dockerfile, when present
  (read from the `# renovate:`-annotated `ARG <APP>_VERSION=` line).

Images are **signed keyless with [cosign](https://docs.sigstore.dev/)** (Sigstore +
GitHub OIDC — no private key to manage). Verify a pull with:

```bash
cosign verify ghcr.io/rubasace/shipofharkinian:edge \
  --certificate-identity-regexp '^https://github.com/rubasace/gow/' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
```

The matrix is generated by
[`.github/scripts/build-matrix.sh`](.github/scripts/build-matrix.sh), which discovers
images, decides which ones to build for the current event, and extracts each version.
It's runnable locally (`GITHUB_EVENT_NAME=workflow_dispatch bash .github/scripts/build-matrix.sh`).

### Version bumps (Renovate)

[`renovate.json5`](renovate.json5) tracks the `ARG <APP>_VERSION=` pins. When upstream
releases, Renovate opens a PR bumping the pin; merging it to `main` triggers a build that
publishes the new `<version>` tag and refreshes `edge`. Auto-merge is intentionally **off**
(an upstream release can break the build); enable it per-package by adding `"automerge": true`
to a Renovate rule if you want fully hands-off updates.

## Adding a new image

Zero CI config — the pipeline is convention-based:

1. Create `images/<name>/` with a `Dockerfile` (and whatever `scripts/`, `configs/`,
   `wolf/` it needs). The folder name **is** the GHCR repository name, so keep it
   lowercase and registry-safe.
2. To get a version tag in addition to `edge`, pin the upstream version with a
   Renovate-annotated build arg — the existing images follow this exactly:

   ```dockerfile
   # renovate: datasource=github-releases depName=<owner>/<repo>
   ARG MYAPP_VERSION=1.2.3
   ```

   The arg name only has to end in `_VERSION`. An image with no such pin still builds
   and gets the rolling `edge` tag.
3. Open a PR / push to `main`. The new folder is detected and built automatically; no
   change to the workflow is needed.

## One-time GHCR setup

On the **first** publish, each new GHCR package is created **private**. To pull without
authenticating (and to match "use the public registry"), open the package on GitHub once
and set **Package settings → Danger Zone → Change visibility → Public**. The package is
linked to this repo automatically (via the `org.opencontainers.image.source` label), so
repo collaborators already inherit access. No other secrets are required — the workflow
uses the built-in `GITHUB_TOKEN`.
